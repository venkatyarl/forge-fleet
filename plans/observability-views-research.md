# Observability views — native replacement for the retired LANG.3/Langfuse plan

**Context.** `lang3-views.md` (3 saved Langfuse/ClickHouse views) was retired
2026-06-22 — Langfuse isn't running, OTLP export is off, and the dead
OTLP→Langfuse code path was removed (this PR). The three views the operator
actually wanted are better served *natively* off `ff_interactions` (the
per-call LLM log) + the fleet tables. This doc researches how to rebuild
View 3 and improve Views 1 & 2 with no external trace store.

## Source of truth: `ff_interactions`
Every routed LLM call is logged here (req + resp + routing + cost). Columns
relevant to the views:
`ts, session_id, channel, user_id, route_decision, engine, worker_name,
endpoint, tokens_in, tokens_out, cost_usd, latency_ms, outcome,
error_signature, ff_build_sha, model_versions`.
This is richer than Langfuse's `observations` ever was — it carries
`outcome`/`error_signature` (success-rate views) and `ff_build_sha`/
`model_versions` (regression attribution) that Langfuse never had.

## View 1 — "work by computer" (was: spans by computer)
**Covered today** by a plain group-by; improve it into a first-class verb +
rollup.

Baseline query (already works):
```sql
SELECT worker_name, count(*) AS calls,
       sum(tokens_in+tokens_out) AS tokens,
       round(avg(latency_ms)) AS avg_ms,
       round(sum(cost_usd)::numeric,4) AS cost_usd,
       max(ts) AS last_seen
FROM ff_interactions
WHERE ts > now() - interval '7 days'
GROUP BY worker_name ORDER BY tokens DESC;
```

### Improvements to research/build
1. **`ff usage --by computer|engine|channel --window 7d`** CLI verb + the
   matching `usage_*` MCP tool, so the operator/agents don't hand-write SQL.
2. **Latency percentiles, not just avg** — `percentile_cont(0.5/0.95)` over
   `latency_ms` per worker (avg hides tail latency; p95 is what matters for a
   wedged Priya-class host).
3. **Success rate per worker** — `count(*) FILTER (WHERE outcome='ok')::float
   / count(*)` — surfaces a node that answers but errors a lot.
4. **Live + historical split** — `ff pulse` already shows *live* per-computer
   state; this view is the *historical* complement. Cross-link them.

## View 2 — "cost split by tier" (was: cost by tier)
**Covered today**; `engine` is the tier proxy (`qwen36-35b`=local vs
`codex`/`kimi`/`claude`=cloud). Improve into the headline ROI metric.

Baseline (already works):
```sql
SELECT engine, count(*) AS calls,
       sum(cost_usd) AS cost_usd,
       sum(tokens_in) AS in_tok, sum(tokens_out) AS out_tok
FROM ff_interactions
WHERE ts > now() - interval '30 days'
GROUP BY engine ORDER BY cost_usd DESC;
```

### Improvements to research/build
1. **Explicit `tier` dimension** — derive `local|cloud` (and `9b/30b/70b`)
   from `engine`/`endpoint` rather than string-matching at query time. Either
   a generated column on `ff_interactions` or a small `engine_tier` lookup.
   This is the single most useful add — it makes "what % went to cloud" a
   one-liner instead of a CASE soup.
2. **Counterfactual cloud cost ("savings")** — the project's whole thesis is
   "replace cloud subscriptions." For each *local* call, compute what the same
   `tokens_in/out` would have cost on a reference cloud model (rate card in a
   `cloud_rate_card` table), and sum `would_have_cost - actual_cost` =
   **$ saved by the fleet**. This is the number worth putting on the dashboard.
3. **Budget tie-in** — `ff_observability::metrics` already exports
   `LLM_BUDGET_*` + `LLM_COST_USD_TOTAL` Prometheus gauges; reconcile this view
   against them so the historical table and the live gauge agree.

## View 3 — fabric saturation (NOT covered — needs a new link)
This is the real build. Today: `fabric_pairs` has the *bandwidth* side
(`measured_bandwidth_gbps`, 3 pairs — 2× CX7, 1× Thunderbolt@20Gbps) and
`fabric_measurements` has iperf history — but **nothing attributes LLM traffic
to a fabric pair**. `ff_interactions` records `worker_name` + `endpoint`, not
"this call's tensor-parallel traffic crossed pair X." (Same gap flagged in the
multi-host-deployment memo: TP=2 vLLM across sia+adele is invisible to pulse.)

### The missing piece: a deployment ↔ fabric_pair link
Fabric traffic is generated by **multi-host inference** (e.g. TP=2 vLLM across
a CX-7 pair). So saturation = (LLM token throughput routed to a TP deployment)
÷ (that pair's measured bandwidth). Two things must exist first:
1. **A deployment knows it spans a fabric pair.** Add `fabric_pair_id` (+ maybe
   `tp_degree`) to the multi-host deployment row (or a
   `deployment_fabric_pairs` junction). This is the V43 multi-host schema the
   deployment memo already calls for — View 3 is a *consumer* of it.
2. **Bytes estimate per call.** `ff_interactions` has tokens; TP traffic ≈
   `tokens × hidden_dim × bytes_per_elem × layers` for activations — too
   model-specific to estimate from tokens alone. **Better: sample real iface
   counters**, not infer from tokens.

### Two viable designs (research which to build)
**(A) Routing-attributed (cheap, approximate).** Join
`ff_interactions.endpoint → deployment → fabric_pair_id`, sum
`tokens_out` per pair per hour, divide by `measured_bandwidth_gbps` for a
*relative* "which pair is hottest" signal. Pros: reuses existing data once the
link exists. Cons: tokens are a poor proxy for actual fabric bytes.

**(B) Counter-sampled (accurate, new collector).** A per-node tick samples the
fabric iface byte counters (`/sys/class/net/<iface>/statistics/{rx,tx}_bytes`
on Linux; `netstat -ib` on macOS) every N seconds, diffs them, and writes
`fabric_saturation_snapshots(snapshot_at, pair_name, bytes_per_sec,
measured_gbps, saturation_pct)`. Pros: real utilization, independent of LLM
attribution. Cons: a new sampler tick (mirror `disk_sampler`), per-iface.

**Recommendation:** **(B) is the correct "saturation" metric** — it measures
the actual link, the way `disk_sampler` measures actual disk. Build it as a
leader-or-per-node `fabric_sampler` tick reusing the spawn-tick pattern, keyed
on `fabric_pairs.{a_iface,b_iface}` per host. Layer (A) on top later if you
want to *attribute* the bytes to specific deployments/models. Until (B) ships,
View 3 stays honestly unbuilt — don't fake it from token math.

### Proposed table (for design B)
```sql
CREATE TABLE IF NOT EXISTS fabric_saturation_snapshots (
    snapshot_at    TIMESTAMPTZ NOT NULL,
    pair_name      TEXT        NOT NULL,
    host           TEXT        NOT NULL,   -- which endpoint sampled
    iface          TEXT        NOT NULL,
    rx_bytes_per_s BIGINT      NOT NULL,
    tx_bytes_per_s BIGINT      NOT NULL,
    measured_gbps  DOUBLE PRECISION,        -- from fabric_pairs at sample time
    saturation_pct DOUBLE PRECISION,        -- (rx+tx)*8/1e9 / measured_gbps
    PRIMARY KEY (snapshot_at, pair_name, host, iface)
);
```

## Suggested sequencing
1. **View 2 tier + savings** (P0) — highest operator value ("how much is the
   fleet saving"), pure SQL + a small rate-card table. No new collectors.
2. **`ff usage` verb + percentiles/success-rate** (P0) — makes Views 1 & 2
   first-class instead of ad-hoc SQL.
3. **`fabric_sampler` tick + `fabric_saturation_snapshots`** (P1, design B) —
   the only genuinely new infra; depends on nothing but `fabric_pairs`.
4. **Deployment↔fabric_pair link** (P2) — unlocks per-deployment attribution
   (design A) and is shared with the broader multi-host deployment schema.

All of this lives in ff's own Postgres + dashboard — no Langfuse, no OTLP,
no external trace store.
